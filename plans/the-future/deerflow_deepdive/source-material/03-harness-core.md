# 03 -- Harness Core: The Agent Runtime

## What It Does

The harness core is DeerFlow's agent runtime. The entry point is `make_lead_agent()` in
`backend/packages/harness/deerflow/agents/lead_agent/agent.py`. Every conversation thread
calls this factory exactly once to produce a LangGraph agent. The factory does six things
in sequence:

1. **Resolves the model name.** Priority: request-level override (`model_name` / `model`
   in the configurable dict) > per-agent config (`load_agent_config(agent_name).model`) >
   global default (first entry in `config.yaml` models list). If the requested model is
   not in the config, it falls back to the global default and logs a warning.

2. **Creates a chat model** via `create_chat_model()`, which uses reflection-based class
   resolution (`resolve_class` from `deerflow/reflection/resolvers.py`) to instantiate the
   provider-specific class stored as a colon-delimited string (e.g.,
   `"langchain_openai:ChatOpenAI"`).

3. **Assembles tools** from multiple sources through `get_available_tools()`.

4. **Builds a middleware chain** of up to 13 stages, ordered by explicit comments in
   `_build_middlewares()`.

5. **Applies a dynamic system prompt** via `apply_prompt_template()`, which injects
   conditional sections for subagent orchestration, memory, skills, deferred tools, and
   clarification rules.

6. **Returns a LangGraph agent** by calling `create_agent()` with `state_schema=ThreadState`.

A separate bootstrap path (`is_bootstrap=True`) produces a minimal agent with reduced
prompt and the extra `setup_agent` tool, used for the initial custom-agent creation flow.

## How It Works

### ThreadState

**File:** `deerflow/agents/thread_state.py`

`ThreadState` extends LangChain's `AgentState` (which already carries `messages`) with
seven additional fields:

| Field | Type | Behavior |
|---|---|---|
| `sandbox` | `SandboxState` (TypedDict: `sandbox_id`) | Passed through to sub-agents |
| `thread_data` | `ThreadDataState` (TypedDict: `workspace_path`, `uploads_path`, `outputs_path`) | Set by ThreadDataMiddleware |
| `title` | `str \| None` | Generated by TitleMiddleware after first exchange |
| `artifacts` | `Annotated[list[str], merge_artifacts]` | Deduplicating reducer using `dict.fromkeys` for order-preserving merge |
| `todos` | `list \| None` | Managed by TodoMiddleware in plan mode |
| `uploaded_files` | `list[dict] \| None` | Populated by UploadsMiddleware |
| `viewed_images` | `Annotated[dict[str, ViewedImageData], merge_viewed_images]` | Merging reducer; sending empty dict `{}` clears all images |

The two `Annotated` reducers (`merge_artifacts`, `merge_viewed_images`) handle LangGraph's
state-update semantics: each graph step returns a partial dict, and the reducer merges it
into the accumulated state.

### Tool Assembly

**File:** `deerflow/tools/tools.py` -- `get_available_tools()`

Tools are assembled in this order:

1. **Config-based tools.** Each `ToolConfig` entry in `config.yaml` has a `use` field
   (a colon-delimited module:variable path). `resolve_variable()` imports and returns the
   tool instance. An optional `groups` filter restricts which tools are loaded for
   custom agents.

2. **Built-in tools.** Always included: `present_file_tool`, `ask_clarification_tool`.
   `view_image_tool` is added only when the resolved model's `supports_vision` is true.

3. **Subagent tools.** `task_tool` is appended only when `subagent_enabled=True`.
   (`task_status_tool` was removed -- backend handles polling internally.)

4. **MCP tools.** Loaded from `get_cached_mcp_tools()` if any MCP servers are enabled
   in `ExtensionsConfig`. MCP tools are read fresh from disk each call (not from the
   cached `config.extensions`) so Gateway API changes take effect immediately.

5. **Deferred tool registry.** When `tool_search.enabled` is true, all MCP tools are
   registered into a `DeferredToolRegistry` instead of being bound to the model directly.
   The `tool_search` tool is then added to builtins so the agent can load specific tools
   on demand.

The final return is `loaded_tools + builtin_tools + mcp_tools`.

### Sub-Agent Delegation

**Files:**
- `deerflow/tools/builtins/task_tool.py` -- the `@tool("task")` function
- `deerflow/subagents/executor.py` -- `SubagentExecutor`
- `deerflow/subagents/config.py` -- `SubagentConfig` dataclass

**task_tool interface:**

```
task(description, prompt, subagent_type="general-purpose"|"bash", max_turns=None)
```

The tool extracts parent context (sandbox state, thread data, thread_id, parent model
name, trace_id) from `ToolRuntime`, then builds a `SubagentExecutor`.

Critically, subagents call `get_available_tools(subagent_enabled=False)` to prevent
recursive nesting -- a subagent can never spawn another subagent.

**SubagentExecutor concurrency model:**

Two global `ThreadPoolExecutor` instances:

- `_scheduler_pool` -- 3 workers, prefix `subagent-scheduler-`. Receives the `run_task`
  closure from `execute_async()`.
- `_execution_pool` -- 3 workers, prefix `subagent-exec-`. The scheduler submits the
  actual `self.execute()` call here with a `Future.result(timeout=...)` wrapper.

`execute_async()` returns a `task_id` immediately. The task tool then polls
`get_background_task_result()` every 5 seconds in a blocking loop, streaming
`task_started`, `task_running`, `task_completed`, `task_failed`, or `task_timed_out`
events via `get_stream_writer()`. A polling safety net caps iterations at
`(timeout_seconds + 60) // 5`.

`execute()` (the sync wrapper) calls `asyncio.run(self._aexecute(...))`, creating a
fresh event loop per invocation. `_aexecute()` uses `agent.astream()` to capture AI
messages in real-time, deduplicating by message ID.

**Constants and defaults:**

- `MAX_CONCURRENT_SUBAGENTS = 3` (module-level constant in `executor.py`)
- `SubagentConfig.timeout_seconds = 900` (15 minutes)
- `SubagentConfig.max_turns = 50`
- `SubagentConfig.disallowed_tools` defaults to `["task"]`
- `SubagentConfig.model` defaults to `"inherit"` (uses parent's model)

### Configuration

**File:** `deerflow/config/app_config.py`

`AppConfig` is a Pydantic `BaseModel` loaded from YAML via `AppConfig.from_file()`.

**Config path resolution** (`resolve_config_path`):
1. Explicit `config_path` parameter
2. `DEER_FLOW_CONFIG_PATH` environment variable
3. `./config.yaml` in CWD
4. `../config.yaml` (parent of CWD)

**Environment variable resolution:** `resolve_env_variables()` recursively walks the
parsed YAML dict. Any string value starting with `$` is replaced with
`os.getenv(value[1:])`. Raises `ValueError` if the variable is not set.

**Config version check:** On load, compares `config_version` in the user's file against
`config.example.yaml` (searched up to 5 parent directories). Emits a warning if outdated.

**Singleton caching:**
- `get_app_config()` -- returns cached instance, creates on first call
- `reload_app_config()` -- reloads from file, replaces the cached instance
- `reset_app_config()` -- clears cache so the next `get_app_config()` reloads
- `set_app_config()` -- injects a custom/mock instance (for tests)

Side-effect loaders during `from_file()`: title, summarization, memory, subagents,
tool_search, and checkpointer configs are each loaded from their respective dicts via
dedicated `load_*_from_dict()` functions that populate their own singletons.

### Model Factory

**File:** `deerflow/models/factory.py` -- `create_chat_model()`

Steps:

1. Resolve config for the named model (or first model if `name` is None).
2. Call `resolve_class(model_config.use, BaseChatModel)` to import the provider class
   via the `"module_path:ClassName"` string.
3. Dump model config to kwargs, excluding metadata fields (`use`, `name`, `display_name`,
   `description`, `supports_thinking`, `supports_reasoning_effort`,
   `when_thinking_enabled`, `thinking`, `supports_vision`).
4. Merge `when_thinking_enabled` settings with the `thinking` shortcut field. If
   `thinking_enabled=True` and the model supports it, apply the merged overrides. If
   `thinking_enabled=False` and thinking settings exist, explicitly disable thinking
   (handling both OpenAI-compatible `extra_body` nesting and native `langchain_anthropic`
   direct parameter).
5. Strip `reasoning_effort` if the model does not support it.
6. Instantiate the model class.
7. If LangSmith tracing is enabled, attach a `LangChainTracer` callback.

### System Prompt

**File:** `deerflow/agents/lead_agent/prompt.py` -- `apply_prompt_template()`

The prompt is a single `SYSTEM_PROMPT_TEMPLATE` string with `{placeholder}` slots filled
at runtime. Conditional sections:

- **Soul** (`{soul}`): Per-agent personality from `SOUL.md` files, wrapped in `<soul>` tags.
- **Memory context** (`{memory_context}`): Injected from `get_memory_data()` with a
  `max_injection_tokens` cap, wrapped in `<memory>` tags. Skipped if memory is disabled.
- **Subagent instructions** (`{subagent_section}`): A 130-line `<subagent_system>` block
  with the `MAX N` concurrency limit baked into examples. Only included when
  `subagent_enabled=True`.
- **Skills listing** (`{skills_section}`): `<skill_system>` block enumerating all enabled
  skills with name, description, and container file path. Instructs the agent to use
  progressive loading (read skill file on demand).
- **Deferred tools** (`{deferred_tools_section}`): `<available-deferred-tools>` block
  listing MCP tool names when `tool_search` is enabled.
- **Clarification rules** (`<clarification_system>`): Hardcoded block enforcing
  "CLARIFY -> PLAN -> ACT" workflow with five mandatory clarification scenarios.
- **Citation requirements** (`<citations>`): Inline `[citation:TITLE](URL)` format plus
  a Sources section at the end of reports.
- **Working directory paths**: Fixed paths for uploads, workspace, and outputs.
- **Current date**: Appended as `<current_date>` tag at the very end.

## Design Decisions

**Why a supervisor pattern instead of v1's explicit graph.**
DeerFlow v1 used a hand-wired LangGraph `StateGraph` with explicit node functions for
researcher, coder, reporter, and router nodes. v2 replaces that with `create_agent()`,
which produces a single reactive agent node that decides tool calls autonomously. The
explicit graph's routing logic is replaced by the system prompt's clarification and
subagent instructions. This eliminates the need to maintain a graph topology when adding
or removing capabilities -- adding a tool or skill requires zero graph changes.

**Why reflection-based resolution instead of explicit imports.**
Model classes and tool instances are stored as `"module:name"` strings in YAML config.
`resolve_class()` and `resolve_variable()` use `importlib.import_module()` to load them
at runtime. This means new LLM providers or tools can be added by editing config alone,
without touching Python code. It also enables optional dependencies -- the resolver
produces actionable `uv add` / `pip install` hints when imports fail, using a
`MODULE_TO_PACKAGE_HINTS` mapping for known providers.

**Why thread pools + `asyncio.run()` instead of pure async.**
Subagent execution runs inside `ThreadPoolExecutor` workers. Each worker calls
`asyncio.run()` to create a fresh event loop, which is required because (a) MCP tools
use async protocols, and (b) the parent agent's event loop cannot be shared across
threads. The dual-pool design (scheduler + execution) separates timeout management from
actual work: the scheduler pool waits on a `Future.result(timeout=N)` while the execution
pool runs the agent, allowing clean timeout handling via `FuturesTimeoutError`.

**Why deferred tool loading via tool_search.**
When dozens of MCP tools are available, binding all of them to the model increases prompt
size and degrades tool-selection accuracy. Deferred loading registers MCP tools in a
`DeferredToolRegistry` and exposes only a `tool_search` meta-tool. The agent sees tool
names in `<available-deferred-tools>` in the system prompt and can load specific tools on
demand. `DeferredToolFilterMiddleware` hides deferred tool schemas from model binding so
they do not consume context tokens.

## Gotchas

**Middleware order is load-bearing.** The comments in `_build_middlewares()` and
`build_lead_runtime_middlewares()` specify the required order. The base chain from
`build_lead_runtime_middlewares()` is:

1. `ThreadDataMiddleware` -- must be first (sets `thread_data` in state)
2. `UploadsMiddleware` -- needs `thread_id` from ThreadDataMiddleware (lead agent only)
3. `SandboxMiddleware` -- needs thread data for sandbox provisioning
4. `DanglingToolCallMiddleware` -- patches missing `ToolMessages` before model sees history (lead agent only)
5. `ToolErrorHandlingMiddleware` -- converts tool exceptions to error `ToolMessages`

Then `_build_middlewares()` appends conditionally:

6. `SummarizationMiddleware` -- if enabled in config
7. `TodoMiddleware` -- if `is_plan_mode=True`
8. `TitleMiddleware` -- generates conversation title
9. `MemoryMiddleware` -- queues conversation for memory update (after title)
10. `ViewImageMiddleware` -- if model supports vision
11. `DeferredToolFilterMiddleware` -- if `tool_search.enabled`
12. `SubagentLimitMiddleware` -- if `subagent_enabled`, truncates excess parallel task calls
13. `LoopDetectionMiddleware` -- detects and breaks repetitive tool-call loops
14. `ClarificationMiddleware` -- **must be last** to intercept clarification requests after all other processing

Reordering will break assumptions. For example, `UploadsMiddleware` at position 2 depends
on `ThreadDataMiddleware` at position 1 having already set the thread_id.

**Sub-agent recursion prevention.** When `task_tool` creates a sub-agent, it calls
`get_available_tools(subagent_enabled=False)`. This ensures the `task` tool is never in
a sub-agent's tool list, preventing infinite delegation chains. The `SubagentConfig`
also defaults `disallowed_tools=["task"]` as a belt-and-suspenders guard.

**`asyncio.run()` creates new event loops per sub-agent.** Each call to
`SubagentExecutor.execute()` spins up a new event loop via `asyncio.run()`. This is
intentional (thread pool workers have no event loop) but means sub-agents cannot share
async resources (connections, caches) with the parent. It also means calling `execute()`
from an already-running async context will raise `RuntimeError` -- the thread pool
isolation prevents this in practice.

**Config singleton can go stale.** `get_app_config()` caches the first load. If config
is modified at runtime (e.g., via Gateway API writing to `config.yaml`), the cached
instance will not reflect changes until `reload_app_config()` is called. MCP tools
sidestep this by reading `ExtensionsConfig.from_file()` directly on each call to
`get_available_tools()`, but model and tool configs use the stale singleton. For hot
config reloads, the Gateway API must call `reload_app_config()` explicitly.
