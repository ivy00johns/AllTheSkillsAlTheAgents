# 02 вҖ” Architecture

## The Orchestrator-Is-Your-Session Model

Overstory's defining architectural choice: **there is no separate orchestrator
process.** Your Claude Code session IS the orchestrator. Everything is driven by:

1. **CLAUDE.md** вҖ” project instructions loaded at session start
2. **Shell hooks** вҖ” `SessionStart` runs `ov prime`, `UserPromptSubmit` runs `ov mail check --inject`
3. **The `ov` CLI** вҖ” 36+ subcommands invoked via Bash tool

This means the orchestrator has the full power of Claude's reasoning, tool use,
and conversation memory. It's not a state machine вҖ” it's a thinking agent that
happens to have orchestration commands available.

## Two-Layer Agent Instruction Model

Every agent spawned by Overstory receives two instruction layers:

### Layer 1 вҖ” Base Definition (the HOW)
Reusable `.md` files in `agents/` at the repo root. These define the agent's
role, capabilities, constraints, workflow, and communication protocol.
They are the same across all projects and all tasks.

Example agent definitions:
- `agents/scout.md` вҖ” read-only exploration (leaf node, depth 2)
- `agents/builder.md` вҖ” implementation (leaf node, depth 2)
- `agents/reviewer.md` вҖ” read-only validation (leaf node, depth 2)
- `agents/lead.md` вҖ” team lead, can spawn sub-workers (depth 1)
- `agents/coordinator.md` вҖ” top-level orchestrator (depth 0)
- `agents/merger.md` вҖ” branch merge specialist (leaf)
- `agents/monitor.md` вҖ” continuous fleet patrol (no worktree)

### Layer 2 вҖ” Dynamic Overlay (the WHAT)
Per-task `CLAUDE.md` generated by `ov sling` and written to the agent's
worktree. This tells the agent:
- Its unique name and task ID
- The spec file to read
- Its file scope (exclusive ownership)
- Its branch name and worktree path
- Its parent agent and depth in the hierarchy
- Quality gate commands to run
- Pre-loaded mulch expertise
- Canopy profile content (if configured)

The overlay is generated by `src/agents/overlay.ts` from an `OverlayConfig`
object. The template lives in `templates/overlay.md.tmpl`.

## SQLite Backbone

Overstory uses four SQLite databases, all with WAL mode and busy timeouts
for safe concurrent access from multiple agent processes:

| Database | Location | Purpose |
|----------|----------|---------|
| `mail.db` | `.overstory/mail.db` | Inter-agent messaging |
| `sessions.db` | `.overstory/sessions.db` | Agent lifecycle + run tracking |
| `events.db` | `.overstory/events.db` | Tool events, timelines, errors |
| `metrics.db` | `.overstory/metrics.db` | Token usage, cost tracking |

All use `bun:sqlite` (synchronous API). WAL mode enables concurrent readers
with a single writer вҖ” critical when 10+ agents are hitting the same databases.

```sql
-- Typical initialization pattern
PRAGMA journal_mode=WAL;
PRAGMA busy_timeout=5000;
```

## Git Worktree Isolation

Each agent gets its own git worktree вҖ” a separate checkout of the repo at a
unique branch. This provides:

- **File isolation** вҖ” agents can't accidentally overwrite each other's work
- **Branch isolation** вҖ” each agent commits to its own branch
- **Git-native** вҖ” no special tooling needed, just `git worktree add`
- **Cleanup** вҖ” `ov worktree clean` removes finished worktrees

Worktrees live under `.overstory/worktrees/{agent-name}/`. The worktree
manager (`src/worktree/manager.ts`) handles creation, listing, and cleanup
via `Bun.spawn(["git", "worktree", ...])`.

## Tmux Session Management

Each interactive agent runs in its own tmux session. The tmux manager
(`src/worktree/tmux.ts`, 685 lines) handles:

- Session creation with runtime-specific spawn commands
- Readiness detection (polling pane content for TUI state)
- Send-keys for initial prompts and nudges
- Capture-pane for health checks and inspection
- Session cleanup on agent completion

For headless runtimes (Codex, Sapling), tmux is bypassed entirely вҖ”
agents run as direct Bun subprocesses with NDJSON stdout.

## CLI Architecture

The `ov` CLI is built with Commander.js. Entry point is `src/index.ts`.
Each subcommand gets its own file in `src/commands/`.

### JSON Output Envelope

All commands support `--json` for machine-readable output. The envelope
is standardized via `src/json.ts`:

```typescript
// Success
jsonOutput({ agents: [...], stats: {...} });
// Error
jsonError("Something failed", { details: "..." });
```

This enables programmatic orchestration вҖ” any command's output can be
consumed by another agent or script.

### Global Flags

Every command inherits:
- `--json` вҖ” structured JSON output
- `--verbose` вҖ” extra detail
- `--quiet` / `-q` вҖ” suppress non-error output
- `--timing` вҖ” print execution time
- `--project <path>` вҖ” target a different project root

## Configuration System

Config is loaded by `src/config.ts`:

1. **Defaults** вҖ” hardcoded sensible defaults
2. **`config.yaml`** вҖ” project-level settings (committed)
3. **`config.local.yaml`** вҖ” machine-specific overrides (gitignored)

The `OverstoryConfig` type defines the full config surface:
- `project` вҖ” name, root, canonical branch, quality gates, default canopy profile
- `agents` вҖ” manifest path, max concurrent, stagger delay, max depth, max sessions
- `worktrees` вҖ” base directory
- `taskTracker` вҖ” backend selection (auto/seeds/beads)
- `mulch` вҖ” enabled, domains, prime format
- `merge` вҖ” AI resolve enabled, reimagine enabled
- `providers` вҖ” model provider configs (native/gateway, base URLs, auth)
- `watchdog` вҖ” tier enablement, intervals, thresholds
- `models` вҖ” per-capability model overrides
- `logging` вҖ” verbose, redact secrets
- `coordinator` вҖ” exit triggers (all agents done, tracker empty, shutdown signal)
- `runtime` вҖ” default adapter, per-capability overrides, print command, Pi config, shell init delay
